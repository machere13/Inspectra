# Расследование: как сайт выдаёт свои секреты через DevTools

> В Inspectra ты — агент. Не зритель, который читает про веб, а тот, кто открывает капот и смотрит, как всё крутится на самом деле. Эта статья — не туториал «нажми сюда», а набор приёмов расследования. Прочитал абзац — сразу проверь на любом сайте в соседней вкладке.

Любая страница, которую ты открываешь, — это не картинка. Это работающая система: дерево элементов, поток сетевых запросов, хранилище состояния и куча кода, который всё это связывает. Браузер уже даёт тебе инструмент, которым пользуются разработчики, тестировщики и охотники за уязвимостями — **DevTools** (`F12` или `Ctrl+Shift+I`, на Mac `Cmd+Opt+I`).

Ниже — четыре «комнаты» расследования. В каждой есть, что найти.

---

## 1. DOM: то, что ты видишь — не то, что прислал сервер

Открой вкладку **Elements** (в Firefox — **Inspector**). Перед тобой живое дерево страницы. Ключевое слово — **живое**: это не исходный HTML с сервера, а текущее состояние после работы JavaScript.

Проверь сам:

- Правый клик по странице → **«Просмотр кода страницы»** (`Ctrl+U`) — это то, что прислал сервер.
- А `F12` → **Elements** — это то, во что превратил страницу браузер.

Часто это два разных документа. Контент, подгруженный после загрузки (лента, комментарии, цены), в исходнике отсутствует — он появляется только в DOM.

### Что искать агенту

- **Скрытые элементы.** Текст бывает на странице, но спрятан стилями: `display: none`, `visibility: hidden`, `opacity: 0`, нулевая высота или вынос за экран (`position: absolute; left: -9999px`). Поиск по DOM (`Ctrl+F` внутри Elements) находит то, что глазами не видно.
- **`data-*`-атрибуты.** Разработчики нередко складывают сюда служебную информацию: `data-user-id`, `data-price`, `data-feature-flag`. Это прямой след того, как приложение думает о данных.
- **Закомментированные блоки.** `<!-- старая версия баннера -->` — археология прямо в разметке.

> **Приём «удалённый DOM».** Элемент может появляться и исчезать (модалка, тултип, выпадашка). Во вкладке Elements нажми правой кнопкой → **Break on → subtree / node removal**. Браузер остановит выполнение в момент, когда узел удаляют, — и ты увидишь, какой код это делает.

## 2. Network: водопад, в котором видно всё

Вкладка **Network** — самое честное место в браузере. Здесь каждый запрос: к API, картинкам, шрифтам, аналитике. Открой Network, поставь галочку **Preserve log** и перезагрузи страницу (`Ctrl+R`). Польётся «водопад».

Колонки, на которые смотрит агент:

| Колонка | О чём говорит |
| --- | --- |
| **Name** | Куда ушёл запрос (часто видно endpoint API) |
| **Status** | `200` — ок, `301/302` — редирект, `401/403` — нет доступа, `404` — нет ресурса, `500` — сервер упал |
| **Type** | `fetch/xhr` — это вызовы API, самое интересное |
| **Size** | `(from disk cache)` / `(memory cache)` — отдалось из кэша, а не с сервера |
| **Time** | Кто тормозит загрузку |

### Фокус на API

Отфильтруй запросы по типу **Fetch/XHR** — останутся обращения к бэкенду. Кликни любой → вкладка **Response**: это «сырые» данные (обычно JSON), на которых строится страница. Часто в ответе приходит **больше**, чем показано в интерфейсе: лишние поля, скрытые флаги, служебные id.

Во вкладке **Headers** того же запроса смотри:

- **Request URL и метод** (`GET` читает, `POST/PUT/PATCH/DELETE` меняют).
- **Заголовки запроса** — там часто живут токены авторизации (`Authorization: Bearer ...`) и `Cookie`.
- **Заголовки ответа** — `Content-Type`, `Cache-Control`, `Set-Cookie`, а также защитные: `Content-Security-Policy`, `Strict-Transport-Security`, `X-Frame-Options`. Их отсутствие — уже зацепка.

> **Приём «повтор запроса».** Правый клик по запросу → **Copy → Copy as cURL**. Получишь готовую команду, которой можно воспроизвести запрос вне браузера. Так тестируют API и проверяют, что будет, если поменять параметр.

## 3. Application: где сайт хранит память о тебе

Вкладка **Application** (в Firefox — **Storage**) — это память сайта на твоей стороне. Три главных хранилища:

- **Cookies.** Маленькие пары ключ-значение, которые браузер сам прикладывает к каждому запросу на домен. Чаще всего здесь — идентификатор сессии. Смотри на флаги:
  - `HttpOnly` — куку **нельзя** прочитать из JavaScript (защита от кражи через XSS). Если её нет у сессионной куки — это слабое место.
  - `Secure` — кука уходит только по HTTPS.
  - `SameSite` — `Lax`/`Strict`/`None`: защита от CSRF, правило «прикладывать ли куку к запросам с чужих сайтов».
- **Local Storage / Session Storage.** Хранилище для JS. Сюда кладут настройки, черновики, иногда — токены (что спорно: в отличие от `HttpOnly`-куки, это доступно любому скрипту на странице).
- **IndexedDB / Cache Storage.** База данных и кэш для офлайн-режима и PWA.

### Что искать агенту

- Чувствительные данные в открытом виде (e-mail, роль, флаги доступа).
- Признаки «фронтенд решает, что тебе можно»: например, в localStorage лежит `isAdmin: false`. Если права проверяются **только** на клиенте — это классическая логическая дыра: серверу всё равно нужно проверять самому.

## 4. Console и Sources: код, который всем управляет

**Console** — это не только место для ошибок (хотя красные строки тоже улика: по ним видно, что у приложения сломалось). Это интерактивный доступ к странице. Несколько честных приёмов:

```js
// Сколько на странице ссылок?
document.querySelectorAll('a').length

// Показать все скрытые поля формы
[...document.querySelectorAll('input[type=hidden]')]
  .map(i => ({ name: i.name, value: i.value }))

// Подсмотреть глобальные объекты приложения
Object.keys(window).filter(k => !k.startsWith('webkit'))
```

Во вкладке **Sources** лежит весь JavaScript. Даже «сжатый» (минифицированный) код можно развернуть кнопкой **`{}`** (Pretty print) и поставить точку останова (breakpoint), чтобы пошагово смотреть, как приложение принимает решения.

> **Граница агента.** Читать, разбирать и понимать чужой сайт через DevTools — это нормально: ты исследуешь то, что браузер и так загрузил тебе. А вот эксплуатировать найденное на чужих системах без разрешения — уже не расследование, а нарушение. В Inspectra мы ломаем **специально подготовленные** сцены и учебные стенды. В реальном мире — только то, что твоё или на что есть явное разрешение (bug bounty, пентест по договору).

## Мини-чеклист расследования

Открой любой сайт и пройди по шагам:

- [ ] Сравни **исходник** (`Ctrl+U`) и **DOM** (Elements) — что дорисовал JS?
- [ ] Найди в DOM хотя бы один **скрытый** элемент или `data-*`-атрибут.
- [ ] В **Network → Fetch/XHR** найди вызов API и открой его **Response**.
- [ ] Проверь, нет ли в ответе **лишних полей**, которых нет в интерфейсе.
- [ ] В **Application → Cookies** посмотри флаги `HttpOnly` / `Secure` / `SameSite` у сессионной куки.
- [ ] Загляни в **localStorage** — нет ли там того, что не должно быть на клиенте?
- [ ] Посмотри **Console** на ошибки — что они выдают об устройстве сайта?

---

## Что дальше

Каждый из этих пунктов — отдельная глубина. DOM ведёт к пониманию рендеринга и доступности. Network — к API, кэшированию и производительности. Cookies и Storage — к аутентификации, сессиям и целому классу уязвимостей (XSS, CSRF, broken access control). Console и Sources — к тому, как вообще живёт код в браузере.

Главный навык агента не в том, чтобы знать, **где** нажать. А в привычке **сомневаться и проверять**: «А что на самом деле прислал сервер? А правда ли эту кнопку нельзя нажать? А где проверяются мои права — здесь или там?» Инструмент у тебя уже открыт. Осталось начать смотреть.

> _Не боится ошибок. Хочет понимать, а не заучивать. Не обязан быть программистом._ — это и есть профиль агента Inspectra.
